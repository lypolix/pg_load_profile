# Алгоритм определения профиля нагрузки PostgreSQL 

Платформа для профилирования нагрузки на PostgreSQL и автоматического определения профиля нагрузки (OLTP, OLAP, write‑heavy, read‑heavy, batch и др.) с возможностью последующего тюнинга базы.


## Запуск

1) Скопировать .env.example в .env
2) Запустить Docker Desktop
3) `docker-compose up --build`
4) Открыть фронтенд: `cd frontend && npm install && npm start`
5) Открыть в браузере: http://localhost:3000

## Первый запуск

⚠️ **ВАЖНО**: Перед запуском тестов нагрузки необходимо инициализировать базу данных!

В интерфейсе дашборда нажмите кнопку **"Init DB"** (оранжевая кнопка справа вверху).

Это создаст тестовые таблицы pgbench (займет 30-60 секунд).

Далее можно выбрать изначальную конфигурацию БД и режим нагрузки.

После этого можно динамически наблдать за метрикми и статистиками, и прослеживать рост нагрузки по графикам.

Нажмите "Настройки". Откроется страница где врчуную можно измениь параметры. А также посмотреть совет по настройке по алгоритму бэкенда и автоматически применить его. Также доступны рекомендации от AI с возможностью настройки одним нажатием кнопки.

## Использование

1. **Выбор конфигурации БД**: Нажмите на одну из кнопок конфигурации (OLTP, OLAP, Write-heavy и т.д.)
2. **Запуск нагрузки**: Выберите тип нагрузки из выпадающего меню "Выбор нагрузки"
3. **Наблюдение за метриками**: График и метрики обновляются каждые 5 секунд


## Состав проекта

Репозиторий состоит из трёх основных сервисов:

- **backend** (Go)
  - REST API для запуска сценариев нагрузки и получения результатов;
  - управление нагрузочными сценариями;
  - сбор и агрегация метрик (в т.ч. из PostgreSQL: `pg_stat_statements`, `pg_stat_activity` и др.);
  - вызов сервиса классификации профиля нагрузки.
- **ml-service** (Python)
  - принимает агрегированные метрики;
  - реализует rule‑based/ML‑алгоритм определения профиля нагрузки (OLTP, OLAP, read‑heavy, write‑heavy, batch, mixed, cold‑scan);
  - возвращает профиль и объяснение (какие правила сработали).
- **postgres**
  - экземпляр PostgreSQL для тестов;
  - инициализация схемы, данных и нагрузочных сценариев SQL‑скриптами.

Все сервисы запускаются через `docker-compose`.

## Основные возможности

- Запуск типовых профилей нагрузки на PostgreSQL:
  - транзакционная нагрузка (OLTP);
  - аналитическая/отчётная (OLAP);
  - read‑heavy;
  - write‑heavy;
  - batch/ETL;
  - смешанная (HTAP/mixed);
  - cold/full‑scan сценарии.
- Сбор метрик:
  - RPS, средняя и p95/p99 латентность endpoint’ов;
  - количество обращений к БД на запрос;
  - распределение чтений/записей;
  - топ «тяжёлых» SQL из `pg_stat_statements`;
  - базовые системные метрики (CPU/память/IO).
- Классификация профиля нагрузки по собранным метрикам и возврат:
  - имени профиля (`OLTP`, `OLAP`, `READ_HEAVY`, `WRITE_HEAVY`, `BATCH`, `MIXED`, `COLD_SCAN`);
  - причин (какие признаки привели к такому выводу);
  - используемых агрегированных метрик.

## Быстрый старт

### Предварительные требования

- Docker и Docker Compose;
- опционально для локальной разработки:
  - Go (для запуска backend без контейнера);
  - Python 3.x (для запуска ml‑service без контейнера).

### Запуск через Docker Compose

В корне репозитория:

docker-compose up --build


После успешного запуска:

- backend: `http://localhost:8000` (порт можно менять в `docker-compose.yml`);
- ml-service: доступен внутри docker‑сети по имени сервиса (и опционально пробрасывается наружу);
- postgres: доступен контейнерам по имени сервиса (и опционально по `localhost:5432`).


### Входные метрики (пример)

- HTTP‑уровень:
- RPS;
- latency p50/p95/p99;
- доля ошибок.
- База данных:
- число SQL‑запросов на один HTTP‑запрос;
- доля `SELECT` vs `INSERT/UPDATE/DELETE`;
- количество долгих запросов (по `total_time`, `rows`, `mean_time` из `pg_stat_statements`);
- наличие full‑scan / крупных JOIN.
- Ресурсы:
- загрузка CPU;
- использование памяти;
- признаки IO‑узких мест.

### Основные профили

- **OLTP** — транзакционная высококонкурентная нагрузка:
- высокий RPS;
- короткие запросы;
- относительно небольшой размер выборок;
- смешанные read/write операции.

- **OLAP** — аналитика/отчёты:
- низкий RPS;
- длинные и тяжёлые SELECT‑запросы;
- большие выборки, сложные планы.

- **READ_HEAVY**:
- доминируют SELECT;
- мало записей;
- хороший cache‑hit.

- **WRITE_HEAVY**:
- высокая доля INSERT/UPDATE/DELETE;
- возможны блокировки и рост latency на коммитах.

- **BATCH / ETL**:
- периодические массовые операции;
- большие батчи, длинные транзакции.

- **MIXED / HTAP**:
- сочетание коротких транзакций и тяжёлых запросов;
- скачки латентности и нагрузки при запуске отчётов.

- **COLD_SCAN / FULL_SCAN**:
- редкие, но очень тяжёлые запросы с почти полным сканом таблиц.

### Rule‑based классификация

Классификатор реализован как набор правил по метрикам:

- если низкий RPS и есть длинные тяжёлые запросы с большими выборками → `OLAP` или `BATCH`;
- если высокий RPS, короткие запросы, умеренный write‑ratio → `OLTP`;
- если write‑ratio > порога и замечены задержки/блокировки → `WRITE_HEAVY`;
- если доминируют SELECT и высокая кэшируемость → `READ_HEAVY`;
- если есть одновременно короткие и очень длинные запросы, а p95/p99 значительно выше p50 → `MIXED`;
- если один/несколько запросов потребляют большую долю общего времени и делают full‑scan → `COLD_SCAN`.
